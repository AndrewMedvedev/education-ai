# Отчет по инженерии программного обеспечения

**Название проекта:** <b>MindMap</b> — сервис интеллект-карт для студентов и команд

## 1. Общий контекст проекта
### Цель проекта:
Разработка веб-приложения для создания и совместного редактирования интеллект-карт (mind maps). Основная цель — упростить процесс визуализации мыслей, конспектирования лекций и мозговых штурмов в распределенных командах.

### Целевая аудитория:
 - **Студенты:** Для структурирования учебного материала, подготовки к экзаменам.
 - **Проектные команды (малые группы до 10 человек):** Для проведения брейнстормов и планирования спринтов.
 - **Преподаватели:** Для создания наглядных учебных пособий.

### Ценность проекта:
Проект решает проблему отсутствия бесплатных, простых в использовании инструментов с функциями реального времени (коллаборации) на русском языке (или с простым порогом входа). В отличие от сложных аналогов (типа Miro), MindMap фокусируется только на одном типе диаграмм, делая интерфейс интуитивным и быстрым.

## 2. Функциональные и нефункциональные требования
### Функциональные требования (Что система должна делать)
 - **Управление картами:** Пользователь должен иметь возможность создавать, переименовывать, удалять и копировать интеллект-карты.
 - **Редактирование узлов:** Пользователь может создавать новые узлы (блоки), редактировать их текст, менять цвет и перемещать их по канвасу (drag & drop).
 - **Коллаборация в реальном времени:** Несколько пользователей одновременно могут редактировать одну карту; изменения одного участника должны мгновенно (с задержкой не более 500 мс) отображаться у других.
 - **Экспорт:** Пользователь может экспортировать готовую карту в форматы PNG и PDF для вставки в отчеты или презентации.
 - **Аутентификация:** Система должна поддерживать регистрацию и вход по email/паролю, а также через сторонние сервисы (Google OAuth).

### Нефункциональные требования (Как система должна работать)
 - **Категория:** Производительность
 - **Время загрузки:** Основной интерфейс редактора должен загружаться не более чем за 2 секунды при средней скорости интернета (10 Мбит/с).
 - **Масштабируемость (нагрузка):** Серверная часть должна выдерживать одновременную работу не менее 100 активных сессий редактирования без падения производительности (на начальном этапе).
 - **Категория:** Безопасность 

Защита данных: Доступ к картам должен быть изолирован: пользователь А не должен видеть карты пользователя Б. Необходимо использовать JWT-токены для проверки прав доступа к каждому API-запросу.
Категория: Удобство сопровождения (Maintainability)

Чистота кода: Проект должен поддерживать покрытие кода тестами не менее 70% (backend) и следовать стандартам линтинга (ESLint для JS, PEP8 для Python, если используется).

Документация: API должен быть задокументирован с помощью Swagger/OpenAPI для упрощения интеграции и понимания структуры бэкенда фронтенд-разработчиком.

## 3. Выбор модели жизненного цикла разработки ПО
Выбранная модель: Agile/Scrum (адаптированный вариант)

Обоснование выбора:

Неполнота требований: Хотя базовый функционал понятен (создание карт), требования к UX (удобству использования) и конкретному набору фич для коллаборации могут меняться по мере получения обратной связи от первых пользователей (студентов).

Итеративный характер: Проект можно разбить на несколько версий (релизов):

MVP (Минимально жизнеспособный продукт): Только локальное редактирование (без сохранения на сервер).

Релиз 1.0: Добавление бэкенда и аутентификации.

Релиз 2.0: Добавление реального времени (WebSockets).
Это идеально соответствует спринтам в Scrum.

Размер команды: Предположим, команда состоит из 2-3 разработчиков. Scrum позволяет гибко распределять задачи на короткие промежутки времени (1-2 недели), что повышает прозрачность процесса.

Обратная связь: Модель позволяет быстро вкатывать изменения, если «заказчик» (в роли которого может выступать преподаватель или тот же студенческий совет) просит поправить интерфейс.

Почему не Waterfall: Потому что требования к интерфейсу и коллаборации сложно полностью специфицировать в начале проекта, не увидев прототип.

## 4. Взаимодействие с ключевыми участниками (Ролевая модель)
В моей команде роли распределены следующим образом (пример описания, если вы работаете в группе):

С заказчиком / Product Owner (преподаватель или куратор проекта):

Как: Проводим демо (review) в конце каждой итерации (спринта).

Зачем: Чтобы убедиться, что мы правильно поняли требования и продукт движется в нужном русле.

С аналитиком (если есть):

Как: Участвуем в уточнении пользовательских историй (user stories). Например, история «Создать узел» обсуждается с аналитиком, чтобы он помог прописать критерии приемки (Acceptance Criteria).

С разработчиками (коллегами по команде):

Как: Ежедневные стендапы (Daily Scrum) по 10-15 минут: что сделал вчера, что буду делать сегодня, какие есть блокеры.

Практики: Проведение code review перед мержем пул-реквестов в основную ветку. Это повышает качество кода.

С тестировщиком (QA):

Как: Передаем задачу в тестирование только после того, как разработчик провел первичное smoke-тестирование. Обсуждаем баги в баг-трекере (например, в Trello или YouTrack), указывая приоритет (Critical, Major, Minor).

С DevOps-инженером:

Как: Согласовываем конфигурацию окружений (dev/stage/prod). Если DevOps отдельного нет, разработчики настраивают CI/CD (GitHub Actions) вместе.
